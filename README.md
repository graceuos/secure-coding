# secure-coding
# Tiny Second-hand Shopping Platform

Secure Coding 과제로 만든 간단한 중고거래 플랫폼입니다. Python Flask로 만들었습니다.

## 실행 방법

1. 저장소 클론
git clone https://github.com/graceuos/secure-coding
cd secure-coding

2. 가상환경 생성 및 활성화
python3 -m venv venv
source venv/bin/activate

3. 필요한 패키지 설치
pip install flask flask-sqlalchemy flask-wtf flask-login flask-socketio

4. 실행
python3 run.py

5. 브라우저에서 `http://localhost:5000` 접속

(참고: 관리자 페이지를 쓰려면 회원가입 후 아래 명령어로 직접 관리자 권한을 줘야 한다.)
python3 -c "from app import create_app, db; from app.models import User; app = create_app(); app.app_context().push(); u = User.query.filter_by(username='본인아이디').first(); u.is_admin = True; db.session.commit()"

## 1. 요구사항 분석

과제 요구사항을 보고 이 플랫폼에 있어야 하는 기능을 아래와 같이 정리했다.

- 회원가입 / 로그인이 가능해야 한다
- 상품을 올리고 볼 수 있어야 한다
- 유저끼리 소통할 수 있어야 한다 (채팅)
- 악성 유저나 상품을 신고하고 차단할 수 있어야 한다
- 유저 간 송금이 가능해야 한다
- 상품 검색이 가능해야 한다
- 관리자가 전체를 관리할 수 있어야 한다

여기에 더해서, 이 과제의 핵심인 "시큐어 코딩"을 적용하는 것이 목표였다. 기능 구현 자체보다, 각 기능에 어떤 보안 문제가 생길 수 있는지 생각하면서 만들려고 했다.

## 2. 시스템 설계

### 데이터베이스 설계

- **User**: id, username, password_hash, bio, report_count, is_active_account, is_admin, balance
- **Product**: id, title, description, price, seller_id, is_blocked, report_count
- **Message**: id, sender_id, receiver_id, product_id, content (product_id가 있으면 상품별 1:1 채팅, 없으면 전체채팅)
- **Report**: id, reporter_id, target_type(user/product), target_id, reason
- **Transaction**: id, sender_id, receiver_id, amount (송금 기록)

비밀번호는 원문이 아니라 해시값(password_hash)으로만 저장하도록 처음부터 설계했다.

### 페이지 구조

- 회원가입 / 로그인 / 마이페이지
- 상품 목록(검색 포함) / 상품 등록 / 상품 상세 / 상품 수정 / 내 상품 관리
- 유저 프로필 조회 / 송금
- 전체 채팅 / 상품별 1:1 채팅 / 내 채팅 목록
- 상품 신고 / 유저 신고
- 관리자 대시보드

원래는 1:1 채팅을 유저 단위로만 설계했는데, 만들다 보니 "어떤 상품에 대해 대화 중인지" 구분이 안 되는 문제가 있어서 상품 단위로 채팅방을 다시 나누도록 설계를 바꿨다.

## 3. 구현

Flask 기반으로 기능별로 파일을 나눠서 구현했다 (auth, products, chat, reports, users, wallet, admin). 각 기능을 만들면서 발견한 보안 문제와 어떻게 고쳤는지는 4번에 정리했다.

기술 스택: Flask, Flask-SQLAlchemy(DB), Flask-Login(로그인 세션), Flask-WTF(폼/CSRF), Flask-SocketIO(실시간 채팅), SQLite

## 4. 개발 중 발견한 보안 약점과 수정 내용

만들면서 "이렇게 짜면 문제가 생기겠다" 싶었던 부분들을 계속 고쳐나갔다. 정리하면 다음과 같다.

**회원가입 / 로그인**
- 비밀번호를 평문으로 저장하면 DB가 유출됐을 때 그대로 노출된다. → `werkzeug.security`의 해시 함수로 저장하도록 수정.
- 로그인 실패 시 "아이디가 없습니다"와 "비밀번호가 틀렸습니다"를 다르게 알려주면, 공격자가 어떤 아이디가 실제로 존재하는지 알아낼 수 있다(계정 열거 공격). → 두 경우 모두 "아이디 또는 비밀번호가 올바르지 않습니다"로 통일.
- 아이디 중복 체크를 서버에서 하지 않으면 같은 아이디로 여러 계정이 생길 수 있다. → 서버 단에서 중복 확인 후 거부하도록 처리.
- 입력값 길이를 클라이언트(브라우저)에서만 검증하면 개발자도구로 우회할 수 있다. → WTForms validators로 서버 측에서도 다시 검증.

**세션 / CSRF**
- 세션 쿠키를 자바스크립트에서 읽을 수 있으면 XSS 공격 시 세션을 탈취당할 수 있다. → `SESSION_COOKIE_HTTPONLY = True`로 설정.
- 로그인, 상품 삭제 같은 상태 변경 요청에 CSRF 방어가 없으면, 사용자가 다른 사이트를 보는 동안 몰래 요청이 전송될 수 있다(CSRF 공격). → Flask-WTF의 CSRFProtect를 앱 전체에 적용하고, 폼이 아닌 삭제 버튼에도 csrf_token을 직접 넣음.

**상품**
- 가격 입력값이 음수이거나 비정상적으로 크면 문제가 될 수 있다. → 서버에서 0 이상, 특정 범위 이내로 검증.
- 다른 사람이 남의 상품을 수정하거나 삭제할 수 있으면 안 된다. → 상품의 seller_id와 현재 로그인한 유저를 비교해서 본인 것이 아니면 403 처리.
- 차단된(신고 누적) 상품이 목록이나 상세페이지에 그대로 노출되면 안 된다. → 목록 조회 시 is_blocked=False만 가져오고, 상세페이지도 판매자 본인이 아니면 404 처리.

**채팅**
- 로그인하지 않은 사용자가 소켓에 연결해서 메시지를 보낼 수 있으면 안 된다. → 소켓 연결(connect) 이벤트에서 로그인 여부 확인 후 아니면 연결 거부.
- 빈 메시지나 지나치게 긴 메시지를 그대로 받아주면 안 된다. → 서버에서 길이와 공백 여부 체크 후 거부.
- 채팅 메시지를 화면에 그대로 넣으면(innerHTML) 사용자가 `<script>` 태그를 메시지로 보내서 다른 사람 브라우저에서 실행시킬 수 있다(XSS). → innerHTML 대신 textContent로 넣어서 태그가 텍스트 그대로만 보이게 처리.

**신고**
- 본인이 등록한 상품이나 본인 계정을 스스로 신고할 수 있으면 신고 기능을 악용할 수 있다. → 본인 대상 신고 시도는 403으로 차단.
- 신고 사유를 안 적어도 신고가 접수되면 무분별한 신고가 늘어날 수 있다. → 신고 사유를 필수 입력으로 설정.
- 신고가 일정 횟수(3회) 누적되면 상품은 자동 차단, 유저는 자동 휴면 처리되도록 구현. 휴면 계정은 로그인 자체가 막힌다.

**검색**
- 검색어를 SQL문에 그대로 이어붙이면 SQL Injection 공격이 가능해진다. → 문자열을 직접 조립하지 않고 SQLAlchemy의 `ilike()` 파라미터 바인딩 방식을 사용해서 자동으로 이스케이프 처리되게 함.

**송금**
- 클라이언트가 보낸 잔액 값을 그대로 믿으면 조작이 가능하다. → 송금 처리 시 서버에서 DB에 저장된 최신 잔액을 다시 조회해서 부족하면 거부.
- 본인에게 본인이 송금하는 것처럼 이상한 요청은 막아야 한다. → 송신자와 수신자가 같으면 400 에러로 차단.

**관리자**
- 관리자 페이지 URL을 안다고 아무나 들어갈 수 있으면 안 된다(Broken Access Control). → 모든 관리자 라우트에 `is_admin` 여부를 확인하는 데코레이터를 걸어서, 관리자가 아니면 무조건 403.

## 5. 테스트

체크리스트를 만들어서 각 기능이 요구사항대로 동작하는지, 위에서 정리한 보안 조치들이 실제로 적용됐는지 하나씩 확인했다.

- 회원가입/로그인 정상 동작, 중복 아이디 가입 시도 시 차단되는지 확인
- 상품 등록/조회/수정/삭제가 본인 것만 가능한지 확인 (다른 계정으로 로그인해서 남의 상품 수정 시도 → 실패 확인)
- 채팅이 실시간으로 전달되는지, 상품별로 채팅방이 분리되는지 확인
- 같은 상품/유저를 계정 3개로 신고했을 때 자동 차단·휴면이 되는지 확인
- 휴면 처리된 계정으로 로그인 시도했을 때 막히는지 확인
- 잔액보다 큰 금액을 송금 시도했을 때 거부되는지 확인
- 관리자가 아닌 계정으로 `/admin` 접근 시 403이 뜨는지 확인
- 검색창에 상품명을 입력했을 때 관련 상품만 나오는지 확인

## 6. 유지보수 / 아쉬운 점

시간 관계상 아래 부분들은 더 다듬지 못했다.

- 화면 디자인이 거의 없어서(CSS 미적용) 사용성은 떨어진다. 기능과 보안 위주로 시간을 썼다.
- 채팅 메시지 수정/삭제 기능은 없다.
- 관리자 페이지가 단순한 표 형태라서, 실제 운영이라면 더 많은 관리 기능(로그 조회 등)이 필요할 것 같다.
- DB 마이그레이션 없이 구조가 바뀔 때마다 DB를 통째로 새로 만들었다. 실제 서비스라면 Flask-Migrate 같은 도구로 데이터를 유지하면서 구조를 바꿔야 한다.

다음에 시간이 더 있다면 프론트엔드를 다듬고, 채팅 파일 첨부 기능이나 알림 기능 등을 추가하고 싶다.
